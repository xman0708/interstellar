# 复杂前台界面的生成式UI (Generative UI) 攻坚方案

当大模型不仅要生成一张图表或一个表单，而是要生成一个带有多级下钻、复杂表单联动、甚至包含看板（Dashboard）的重量级页面时，传统的“大模型直接吐 JSON Schema 渲染”会面临极大的挑战：模型上下文不仅容易爆表，而且推理出来的 Schema 非常容易出错。

在复杂的企业级表单和中后台系统中，我们不能单纯指望“完全无中生有”，必须采用**混合策略 (Hybrid Strategy)** 与 **渐进式动态编排**。

---

## 1. 复杂交互为什么难用 AI 一次性生成？

*   **表单跨字段联动 (Form Linkage)**：“如果选择了A，B 字段才显示且必填；如果 B 字段金额大于 1 万，触发 C 字段的级联菜单”。这种涉及运行时的 JavaScript 副作用和事件冒泡逻辑，极难仅仅通过静态的 JSON Schema 来表达。
*   **状态的长期保持 (State Management)**：带有分页的复杂表格、抽屉详情页。
*   **Token 受限与延迟**：一次性生成包含几百个字段配置的超级深层 JSON 会耗时极长，体验极差。

---

## 2. 破局之道：三种应对复杂 UI 的架构模式

### 模式一：“分形”与递归树级联渲染 (Fractal UI / Recursive Schema)
不要试图让 Agent 一次性把整个复杂页面的状态吐出来。
*   **实现机制**：最开始的 Schema 是一级骨架：“左侧导航 + 顶部过滤栏 + 下方内容区”。
*   当用户在左侧高亮某个条目时，**只向后端 Agent 索要“当前内容区”的局部变动 Schema**，而保留其他布局的状态不动。
*   **前端代码要求**：前端用来解析 JSON 的引擎必须支持树形的递归渲染，并利用 Vuex/Pinia 把“当前页面的完整 Schema 树”作为全局状态管理起来，Agent 返回的只是一棵子树的“补丁 (Patch)”。

### 模式二：Micro-Frontends & Shadow App (微前端级的沙箱热插拔)
面对需要极其复杂的代码逻辑（如包含强交互的拖拽看板、自定义校验逻辑的 50 个字段的大型合同），放弃 JSON Schema，回归“跑代码”。
*   **实现机制**：
    *   Agent (如 Java 后端) 生成的不只是数据，而是一整包用 JS 写的、打包好的 Vue 组件资产。
    *   利用如 `qiankun` (微前端) 或 `Web Components` 标准，前端主干在运行时通过 `<link>` 或 iframe 动态挂载这一整陀自带交互逻辑的业务包。
*   **这其实是前文提到的 WebAssembly/Docker 沙箱在前端的平替**。前端提供一个安全的“影子应用 (Shadow App)”容器，里面跑 AI 当场写的 JS 逻辑，就算崩溃或者变量污染，也只在那个隔离环境里。

### 模式三：固定视图与生成视图的“混合模式”(Hybrid Mode) — 🌟 企业级极力推荐
不要教条主义地认为“所有 UI 都必须是 AI 生成的”。在绝大多数企业项目（如 HR 系统、合同系统）里，80% 的复杂交互都是确定性的，只有 20% 的展示层需要千人千面。
*   **实现机制**：
    *   最核心、最复杂的流程（如复杂的审批流配置器、图文并茂的长表单收集）仍然是由人类前端工程师硬编码 (Hard-coded) 写死的 Vue `*.vue` 组件。
    *   但是，**这个组件会对外暴露出插槽 (Slots) 和极度丰富的 Props**。
    *   大模型的 Schema 不负责布局页面，而是充当“高级数据喂养员”。大模型只输出 `action: openApprovalModal, params: { formData: ... }`。前端收到指令，打开提前写好的超级复杂对话框。

## 3. 对 `openClaw` 和现有中后台项目的启示

如果你正在重构或为一套中后台加入生命体特征，请遵循以下渐进式推进路线：

1.  **L1 阶段（只读生成器）**：最开始，大模型生成的 UI Schema 只能是**“只读性质的”**（如图表展示、对比表格、数据清单）。一旦用户试图“操作”这些数据，大模型仅返回一条“操作栏配置”，点击后打开的是前端手写的、固定的、负责实际发请求的常规组件。
2.  **L2 阶段（轻表单生成器）**：大模型开始尝试生成带有输入框、单选、多选的轻量级表单 Schema 用于信息收集。遇到表单复杂联动，不让 LLM 现写逻辑，而是我们在工程里预置好一套“基于规则引擎的联动脚本”，大模型只需吐出对应的规则 ID 或极简表达式。()
3.  **L3 阶段（组件热分发）**：使用 Micro-Frontend 技术，真正把一坨包含所有业务逻辑的前端代码动态注入到用户的浏览器里。

**总结：面对复杂界面，不要逼迫 JSON Schema 表达运行时的强逻辑。要么采用微前端动态拉取 JS 执行代码，要么采用“手写复杂外壳 + AI 智能填充心智参数”的混动路线。**
